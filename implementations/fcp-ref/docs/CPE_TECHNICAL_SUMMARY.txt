OPENAI
────────────────────────────────────────────────────────────────────────────
API URL:        https://api.openai.com/v1/chat/completions
API Version:    2024 (✅ Current)
Default Model:  gpt-4o
Max Tokens:     8192
Base URL Env:   OPENAI_BASE_URL (supports OpenAI-compatible endpoints)

Tool Calling:
  Format:       message.tool_calls[].{id, function:{name, arguments}}
  Input Type:   JSON string (REQUIRES PARSING)
  Support:      ✅ Native, tool IDs present, parallel calls
  
Response Fields:
  stop_reason:  finish_reason ("tool_calls" | "stop")
  tokens:       usage.{prompt_tokens, completion_tokens}

Issues:
  - System message injected in every request (token waste ~100/req)
  - Arguments come as JSON strings (requires parsing + error handling)
  - Silent fallback to empty dict on parse failure
  - Action: Implement prompt caching (P2) + add logging (P0)

GOOGLE
────────────────────────────────────────────────────────────────────────────
API URL:        https://generativelanguage.googleapis.com/v1beta/models
Endpoint:       /{model}:generateContent
API Version:    v1beta (🟡 EXPERIMENTAL - may have breaking changes)
Default Model:  gemini-2.0-flash
Max Tokens:     8192

Tool Calling:
  Format:       candidates[0].content.parts[].{functionCall:{name, args{}}}
  Input Type:   dict (native)
  Support:      ⚠️ NO TOOL IDs - order-dependent mapping
  
Response Fields:
  stop_reason:  finishReason ("STOP" | "TOOL_CALLS" | etc)
  tokens:       usageMetadata.{promptTokenCount, candidatesTokenCount}

Quirks:
  - NO TOOL CALL IDs (Google API limitation)
  - Assumes order preserved (fragile)
  - Requires _last_model_parts/_last_function_calls state
  - Tool results sent as functionResponse parts (not user messages)
  - Ignores "thought" parts (extended thinking)
  
Issues:
  - Order-dependent mapping can break if results reordered
  - Stateful adapter (bad design smell)
  - API is v1beta (experimental, may change)
  - Action: Add synthetic IDs (P0) + document limitation

OLLAMA
────────────────────────────────────────────────────────────────────────────
API URL:        http://localhost:11434/api/chat
API Version:    Latest (✅ Current, local only)
Default Model:  llama3.2
Max Tokens:     8192
Base URL Env:   OLLAMA_BASE_URL

Tool Calling:
  Format:       message.tool_calls[].{function:{name, arguments}}
  Input Type:   dict OR JSON string (🟡 INCONSISTENT)
  Support:      ⚠️ NO TOOL IDs - order-dependent mapping
  
Response Fields:
  stop_reason:  done_reason ("stop" | "tool_calls")
  tokens:       {prompt_eval_count, eval_count}

Quirks:
  - Arguments can be dict OR JSON string (normalize both)
  - NO TOOL CALL IDs (Ollama API limitation)
  - Tool results converted to role:tool messages (not user messages)
  - No streaming support (always stream=False)
  - Has is_available() check
  
Issues:
  - Arguments format inconsistent (dict vs string)
  - Order-dependent mapping (like Google)
  - No streaming (high latency for long outputs)
  - Action: Investigate format inconsistency (P1) + add logging (P0) + streaming (P2)

PAIRING
────────────────────────────────────────────────────────────────────────────
Architecture:   Filesystem-based bridge to external agent
Directory:      ~/.fcp/pairing/<session-id>.{request,response,meta}.json
Model:          "external" (cosmetic only)

Tool Calling:
  Format:       tool_use_calls[].{id?, tool, input{}}
  Input Type:   dict (passed through)
  Support:      ✅ Depends on external agent
  
Protocol:
  1. PairingAdapter creates .meta.json (session_id, key)
  2. Fires on_prompt_pending hook
  3. Writes .request.json (system, messages, tools)
  4. **BLOCKS** up to 300s waiting for .response.json
  5. Parses response → CPEResponse
  6. Deletes .response.json

External Agent Flow:
  1. Connects to fcp_mcp_server via IDE/CLI
  2. Polls .request.json (via fcp_poll tool)
  3. Processes prompt
  4. Writes .response.json (via fcp_respond tool)
  5. PairingAdapter unblocks and returns response

Issues:
  - Filesystem polling not ideal (latency, race conditions)
  - Timeout 300s may be too long
  - Poll interval 0.25s (many wakeups)
  - No built-in retry on file errors
  - Action: Document limitations (P1) + consider inotify (future)

================================================================================
ERROR HANDLING
================================================================================

HTTP Errors (via _http.py):
  401 → CPEAuthError
  429 → CPERateLimitError
  Other → CPEError (with status code)

Parse Errors:
  Anthropic: None (direct dict)
  OpenAI: json.loads() → fallback to empty dict (⚠️ SILENT)
  Google: None (direct dict)
  Ollama: json.loads() → fallback to empty dict (⚠️ SILENT)
  Pairing: Depends on external agent

Recommendation:
  Add logger.warning() on all parse failures
  So developers can debug: "Tool received empty args because..."

================================================================================
TOOL RESULT HANDLING
================================================================================

Format sent by FCP:
  user message with content = "[fcp_exec] {json}\n[fcp_mil] {json}"

How each adapter handles:

  Anthropic: Passes as user text (no special handling)
  OpenAI: Passes as user text (no special handling)
  Google: Parses & converts to functionResponse parts (special)
  Ollama: Parses & converts to role:tool messages (special)
  Pairing: Passes as user text (external agent decides)

Key: Google/Ollama convert format; Anthropic/OpenAI pass through
